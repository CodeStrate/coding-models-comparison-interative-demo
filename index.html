Codestral passes a plain int as the timeout — aiohttp expects aiohttp.ClientTimeout. This will raise ValueError at runtime. Gemma and Devstral both use the correct object constructor.

Why Gemma's Fix Loses Its Purpose

By locking the entire function body, Gemma's version makes every thread wait — including during the potentially slow fetch_fn call. If the fetch takes 5 seconds, a thousand threads would queue up serially. The cache's entire benefit vanishes. Double-checked locking (Devstral/Codestral) avoids this by making cache hits fast (no lock) and only serializing the actual fetch.

The Clever Explanation Gemma Gave

Gemma's Thread A/B walkthrough was genuinely the clearest of all three models. It walked through each step: "Thread A checks cache, misses, acquires lock, fetches. Thread B checks, misses, waits for lock..." The explanation quality was top-tier — the fix just didn't match the explanation's precision.

🔍 Key Takeaways

⚖️

Size vs. Reasoning

Gemma at ~4.5B effective parameters matching 8/10 against 22–24B models is extraordinary. But when the task demands precision under pressure (concurrency fixes), parameter scale still matters. Gemma produced confident but incorrect fixes — a warning against trusting verbose code blindly.

🎯

FIM vs. Agentic Tasks

Codestral's 95.3% FIM benchmark meant nothing on reasoning-heavy tasks. Pattern matching ≠ understanding. If your use case is IDE autocomplete, Codestral is valid. For agentic refactoring, Devstral is in a different league.

🔬

Runtime Correctness

Codestral's aiohttp timeout bug would crash at runtime. Gemma's concurrency "fix" would silently degenerate to serial execution. Devstral was the only model to ship correct code on every test. Production code needs Devstral's precision.

⚡

Conciseness Is a Proxy for Understanding

Devstral consistently got more done with less code. Gemma's over-engineering (unnecessary constants, demo functions, unnecessary async decorators) signals it's compensating for uncertainty with volume. Devstral's tighter answers signal genuine understanding.

🧪

Edge Case Sensitivity

Gemma was the only model to handle the case where yaml.safe_load returns None. This is the kind of defensive programming that prevents production outages. Devstral missed it here but compensated with broader async correctness.

✅

The Verdict on Tool Choice

For production agentic coding: Devstral. For IDE autocomplete/FIM: Codestral. For edge-case detection in small model: Gemma E4B (but always test its output — confidence ≠ correctness).

The Real Answer

No model is perfect. But Devstral is the only one in this set that delivers correct reasoning + working code consistently. Gemma punches well above its weight at ~4.5B parameters, but its concurrency fix demonstrates that confidence can mask incorrectness. Codestral is a specialist — great at autocomplete, unreliable at reasoning.

Devstral — Primary

Best for: Agentic coding, refactoring, async logic

Use Devstral when you need code that works the first time. Its reasoning precision on concurrency and async correctness makes it the safest choice for production systems.

Gemma E4B — Watchlist

Best for: FIM, edge-case detection, low-resource deployment

Remarkable efficiency. Ideal where you need good-but-not-perfect code with minimal parameters. Always code-review or test Gemma's output — its verbosity can mask logic errors.

Codestral — Specialist

Best for: IDE autocomplete, FIM completion, pattern matching

Trust Codestral for fill-in-middle tasks and architectural scaffolding. Don't trust it for async correctness, error handling, or any task requiring runtime precision.
